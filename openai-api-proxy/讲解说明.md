## 模块概览

`openai-api-proxy` 是一个用于 **代理 OpenAI 接口的轻量级 HTTP 网关服务**，主要职责是：

- 对外暴露与 OpenAI 兼容的 `/v1/*` HTTP 接口。
- 在入口处做统一的 **鉴权**（使用自定义 AccessToken），只允许可信调用方访问。
- 在内部将请求 **转发到真实的 OpenAI（或兼容）接口**，并在多 API Key 之间做简单的随机负载分担。
- 提供基础的 **限流** 和 **健康检查** 能力。

在整个系统中，可以将前端或后端的 OpenAI 请求统一指向本模块，由它来控制上游 API 的调用与密钥管理。

## 目录结构与职责说明

- **根目录**
  - `main.go`：服务启动入口。
    - 解析命令行参数（`--config=dev.config.yaml`）。
    - 加载配置（`pkg/config`），根据配置设置 Gin 运行模式。
    - 初始化日志（`pkg/log`）。
    - 创建 Gin 引擎并调用 `routers.InitRouters` 注册路由与中间件。
    - 监听在 `Http.Host:Http.Port` 上提供 HTTP 服务。
  - `dev.config.yaml`：开发环境配置文件，包含：
    - `http`：服务监听地址、端口、访问 Token、Gin 模式。
    - `chat`：上游 OpenAI 的 `base_url` 与多个 `api_keys`。
    - `log`：日志输出路径与日志级别。
  - `Dockerfile`：将本服务打包成容器镜像的构建脚本。
  - `README.md`：基础使用说明（如启动方式、配置示例等）。

- **`pkg/config/`**
  - `config.go`：
    - 使用 Viper 从配置文件加载配置，并映射到：
      - `Http`：
        - `Host`：监听地址（如 `0.0.0.0`）。
        - `Port`：监听端口（如 `8081`）。
        - `AccessToken`：用于下游服务访问本代理的「入口令牌」。
        - `Mode`：Gin 运行模式（`debug` / `release` 等）。
      - `Chat`：
        - `APIKeys`：一组用于访问上游 OpenAI 的 API Key。
        - `BaseURL`：上游 OpenAI API 基础地址（如 `https://api.openai.com/v1`）。
      - `Log`：日志级别与日志文件路径。
    - 提供 `InitConfig` / `GetConfig` 作为全局配置访问入口（单例模式）。

- **`pkg/log/`**
  - `log.go` 及配套文件：
    - 基于 `logrus` 封装的日志组件：
      - `ILogger` 接口定义统一的日志操作方法。
      - `NewLogger` 创建新的 Logger 实例。
      - 包级函数 `SetLevel`、`SetOutput`、`SetPrintCaller` 等设置全局日志行为。
    - 在 `main.go` 中统一初始化：
      - 将日志输出到支持滚动的 Writer。
      - 设置级别与是否输出调用者信息。

- **`routers/`**
  - `routers.go`：
    - `InitRouters(r *gin.Engine)`：
      - 注册健康检查路由：`GET /health`。
      - 为后续路由注册统一中间件：
        - `middleware.Auth()`：鉴权与 API Key 注入。
        - `middleware.RateLimit(10, 10)`：简单限流。
      - 调用 `initProxyRouter` 初始化真正的代理路由。
    - `initProxyRouter`：
      - 创建 `proxy.NewProxy()` 实例。
      - 注册 `/v1/*relativePath` 路由组：
        - 通过 `v1.Any("/*relativePath", p.ChatProxy)` 实现对 `/v1` 下所有方法与路径的统一代理。

- **`proxy/`**
  - `proxy.go`：
    - `Proxy` 结构体：
      - `upStreamUrl`：上游 OpenAI 的基础 URL。
      - `upStream`：`httputil.ReverseProxy` 实例，用于反向代理转发。
    - `NewProxy()`：
      - 从配置中读取 `Chat.BaseURL`，去掉末尾 `/v1`，解析为 `url.URL`。
      - 创建 `httputil.NewSingleHostReverseProxy(upStreamUrl)`。
    - `ChatProxy(ctx *gin.Context)`：
      - 将 `ctx.Request.Host` 设置为上游主机名。
      - 调用 `proxy.upStream.ServeHTTP(ctx.Writer, ctx.Request)`，实现透明转发。
    - 通过这一层，`openai-api-proxy` 实现了对 `/v1/*` 请求的统一反向代理。

- **`middleware/`**
  - `auth.go`：
    - `Auth()` 中间件：
      - 从配置读取入口 AccessToken（`Http.AccessToken`）。
      - 校验请求头中的 `Authorization` 是否等于 `Bearer <AccessToken>`。
        - 不匹配则返回 `401 Unauthorized`。
      - 随机从配置中的 `Chat.APIKeys` 列表中选择一个上游 API Key：
        - 使用 `math/rand` 按当前时间播种。
        - `rand.Intn(len(cnf.Chat.APIKeys))` 获取随机索引。
      - 将请求头中的 `Authorization` 替换为选中的上游 API Key（`Bearer <apiKey>`）。
      - 继续执行后续 handler。
    - 通过该中间件，实现了：
      - 对调用方的统一鉴权（只能持有入口 AccessToken 的调用方才可访问）。
      - 对上游 OpenAI API Key 的简单轮询 / 随机分配。
  - `rate_limit.go`：
    - `RateLimit(r rate.Limit, b int)` 中间件：
      - 使用 `golang.org/x/time/rate` 创建一个限流器。
      - 当 `limiter.Allow()` 返回 `false` 时，直接返回 HTTP 429（Too Many Requests）。
      - 用于保护上游服务与本代理避免被瞬时高并发压垮。

- **`health/`**
  - `health.go`：
    - `Health(ctx *gin.Context)`：
      - 当前实现为空 handler，仅用于 K8s / Docker 等环境的存活检查。
      - 可按需扩展成返回 JSON 或检查上游连通性的健康接口。

## 工作流程（请求链路）

一次典型的请求处理流程如下：

1. 调用方（例如 `ai-chat-service` 或其他后端）将 OpenAI 请求改为指向本代理：
   - 原始：`https://api.openai.com/v1/...`
   - 现在：`http://<proxy-host>:<proxy-port>/v1/...`
   - 请求头中携带 `Authorization: Bearer <Http.AccessToken>` 作为入口令牌。
2. 请求进入 Gin：
   - 首先命中全局中间件链：`Auth()` → `RateLimit(...)`。
3. `Auth()` 中间件：
   - 检查入口 `Authorization` 是否与配置中的 AccessToken 匹配。
   - 随机选取一个上游 `APIKey`，替换当前请求的 `Authorization` 为 `Bearer <APIKey>`。
4. `RateLimit()` 中间件：
   - 判断当前速率是否允许访问，若超限返回 429。
5. 路由匹配：
   - 匹配 `/v1/*relativePath`，交由 `Proxy.ChatProxy` 处理。
6. `Proxy.ChatProxy`：
   - 修改 `ctx.Request.Host` 为上游主机名。
   - 使用 `httputil.ReverseProxy` 将请求转发给上游 OpenAI 服务。
7. 上游响应返回后，`ReverseProxy` 会将响应原样回写给调用方。

通过这一流程，调用方完全可以像直接访问 OpenAI 一样使用 `/v1/...` 接口，同时由本代理统一管理密钥、入口鉴权以及限流。

## 使用到的主要设计模式

- **代理模式（Proxy）**
  - 整个模块的核心就是对上游 OpenAI 接口的代理：
    - 对调用方暴露相同的 `/v1/*` 接口。
    - 在中间添加访问控制、API Key 选择等逻辑。

- **中间件 / 装饰器模式（Decorator）+ 责任链模式（Chain of Responsibility）**
  - Gin 中间件 `Auth` 与 `RateLimit` 在请求到达实际处理函数前后做额外工作，形成一条过滤链。

- **单例模式（Singleton）**
  - `pkg/config` 与 `pkg/log` 使用包级变量提供全局唯一的配置与日志实例。

## 本地启动与调试步骤

1. 进入模块目录：

   ```bash
   cd openai-api-proxy
   ```

2. 安装依赖：

   ```bash
   go mod tidy
   ```

3. 根据需要编辑 `dev.config.yaml`，确保：
   - `http.host` / `http.port` 设置为你期望的监听地址与端口。
   - `http.access_token` 设置为下游服务调用本代理时需要携带的 Token。
   - `chat.base_url` 为上游 OpenAI 兼容接口的地址，例如：
     - `https://api.openai.com/v1`
   - `chat.api_keys` 填写一组有效的上游 API Key。

4. 启动服务：

   ```bash
   go run ./main.go --config=dev.config.yaml
   ```

5. 测试请求示例（以 `curl` 为例）：

   ```bash
   curl -H "Authorization: Bearer <http.access_token>" \
        -H "Content-Type: application/json" \
        -d '{"model":"gpt-3.5-turbo","messages":[{"role":"user","content":"hello"}]}' \
        http://<host>:<port>/v1/chat/completions
   ```

   - 本代理会：
     - 校验入口 Token。
     - 随机选择一个上游 API Key。
     - 将请求转发到 `chat.base_url` 对应的 `/v1/chat/completions`。

## 适合新手的阅读顺序建议

1. `dev.config.yaml`：先大致看懂有哪些配置项（HTTP、Chat、Log）。
2. `main.go`：理解整个服务如何启动、如何使用配置初始化 Gin 与日志。
3. `routers/routers.go`：看路由与中间件是如何注册的。
4. `middleware/{auth.go, rate_limit.go}`：理解鉴权与限流的具体逻辑。
5. `proxy/proxy.go`：理解 `ReverseProxy` 是如何构建以及如何转发 `/v1/*` 请求的。
6. 有需要时再阅读 `pkg/log` 的实现细节，了解如何接入更复杂的日志需求。

掌握这些内容之后，你就可以：

- 安全地将系统中的 OpenAI 调用都接入到本代理中，统一管理 API Key 与访问控制。
- 根据业务需求扩展中间件，例如增加请求日志、指标采集（Prometheus）、更复杂的限流与熔断等能力。
