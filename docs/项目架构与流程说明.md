# AI Chat 项目架构与流程说明文档

## 一、项目概述

本项目是一个**完整的 AI 聊天系统**，采用微服务架构设计，实现了类似 ChatGPT 的智能对话功能。项目采用前后端分离的设计，前端提供用户界面，后端通过多个微服务协同工作，实现与大语言模型的交互、敏感词过滤、上下文管理、聊天记录存储等功能。

该系统的核心特性包括：支持流式响应以提供类似打字机的用户体验，具备敏感词过滤和关键词提取能力，能够管理多轮对话上下文，支持聊天记录的持久化存储和相似问题检索，同时提供了完善的鉴权和限流机制。

## 三、模块详细说明

### 3.1 ai-chat-web（前端应用）

**技术栈**：Vue 3 + TypeScript + Vite + Pinia + Naive UI + Tailwind CSS

**主要职责**：为用户提供类似 ChatGPT 的聊天界面，包括消息展示、输入处理、会话管理、导出功能等。

**核心功能**：

- **聊天界面**：支持 Markdown 渲染、代码高亮、公式显示（KaTeX）
- **流式响应**：通过 `onDownloadProgress` 实时获取服务器推送的增量数据，实现边生成边显示
- **会话管理**：支持多会话创建、切换、删除、清空
- **上下文开关**：可选择是否携带历史对话上下文
- **Prompt 模板**：支持 `/` 开头的快捷指令联想
- **导出功能**：可将聊天记录导出为图片
- **多语言**：支持简体中文、繁体中文、英文
- **主题切换**：支持浅色/深色主题
- **用户认证**：支持短信验证码登录

**目录结构**：

```

│   ├── main.ts                 # 应用入口
│   ├── App.vue                  # 根组件

└── package.json
```

### 3.2 ai-chat-backend（HTTP 网关 / BFF）

**技术栈**：Go + Gin + gRPC

**主要职责**：作为前端与后端服务之间的桥梁，将前端 HTTP 请求转换为 gRPC 调用，并负责鉴权、限流、CORS 等网关功能。

**核心功能**：

- **协议适配**：将前端 HTTP + JSON 请求转换为 gRPC 请求
- **流式转发**：接收 gRPC 流式响应，转换为 HTTP chunked 响应给前端
- **CORS 支持**：处理跨域请求
- **访问鉴权**：验证用户 Token（可选）
- **请求限流**：防止恶意请求
- **静态托管**：直接托管前端构建产物

**接口列表**：

| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/api/chat-process` | POST | 流式聊天接口（核心） |
| `/api/chat` | POST | 一次性聊天接口 |
| `/api/config` | POST | 获取后端配置 |
| `/api/session` | POST | 获取会话状态 |
| `/api/health` | GET | 健康检查 |

### 3.3 ai-chat-service（核心后端服务）

**技术栈**：Go + gRPC + Redis + MySQL + 向量数据库

**主要职责**：处理聊天核心业务逻辑，包括对话管理、大模型调用、敏感词过滤、聊天记录存储等。

**核心功能**：

- **对话管理**：基于 Redis 实现多轮对话上下文，支持上下文开关和 TTL
- **敏感词校验**：调用 `keywords-filter` 服务检测用户输入是否包含敏感词
- **关键词提取**：提取用户输入中的关键词，用于向量检索
- **向量检索**：在向量数据库中查找相似问题，尝试命中标准答案
- **聊天记录**：将问答对持久化到 MySQL
- **流式响应**：支持边生成边推送的流式输出
- **分词计算**：调用 tokenizer 服务计算 token 数，控制上下文长度

**gRPC 接口定义**（参见 `proto/chat.proto`）：

```protobuf
service Chat {
  // 非流式聊天
  rpc ChatCompletion(ChatCompletionRequest) returns (ChatCompletionResponse);
  // 流式聊天
  rpc ChatCompletionStream(ChatCompletionRequest) returns (stream ChatCompletionStreamResponse);
}
```

### 3.4 keywords-filter（敏感词/关键词过滤服务）

**技术栈**：Go + gRPC

**主要职责**：提供文本敏感词校验和关键词提取功能，确保输出内容合规。

**核心功能**：

- **敏感词校验**（Validate）：检测文本是否命中敏感词，返回命中的词
- **关键词提取**（FindAll）：提取文本中所有在词库中的关键词
- **词库管理**：支持从文件加载敏感词/关键词词库

**gRPC 接口定义**：

```protobuf
service Filter {
  // 敏感词校验
  rpc Validate(FilterReq) returns (ValidateRes);
  // 关键词提取
  rpc FindAll(FilterReq) returns (FindAllRes);
}
```

**启动命令**：

```bash
go run ./filter-server/main.go --config=dev.config.yaml --dict=sensitive-dict.txt
```

### 3.5 openai-api-proxy（OpenAI API 代理）

**技术栈**：Go + Gin

**主要职责**：代理 OpenAI API 请求，提供统一的访问入口、鉴权和负载均衡。

**核心功能**：

- **接口兼容**：对外暴露与 OpenAI 兼容的 `/v1/*` HTTP 接口
- **访问鉴权**：使用自定义 AccessToken 验证调用方身份
- **密钥轮询**：在多个 API Key 之间随机分配请求
- **限流保护**：基于令牌桶算法限制请求速率

**工作流程**：

1. 调用方携带入口 AccessToken 请求本代理
2. 代理验证 Token 后，随机选择一个上游 API Key
3. 将请求转发到真实 OpenAI 接口
4. 响应原样返回给调用方

**启动命令**：

```bash
go run ./main.go --config=dev.config.yaml
```

## 四、完整请求处理流程

以下是一次完整的用户聊天请求的处理流程：

### 第一步：前端发起请求

用户在 `ai-chat-web` 的聊天界面输入消息并发送。前端通过 `fetchChatAPIProcess` 函数发起 POST 请求到 `/api/chat-process`：

```javascript
// 前端请求示例
fetch('/api/chat-process', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer <token>'
  },
  body: JSON.stringify({
    prompt: '用户输入的消息',
    options: {
      parentMessageId: '上一次的消息ID'
    }
  })
})
```

前端使用 `onDownloadProgress` 回调处理流式响应：

```javascript
onDownloadProgress: ({ event }) => {
  const xhr = event.target
  const { responseText } = xhr
  const lastIndex = responseText.lastIndexOf('\n', responseText.length - 2)
  let chunk = responseText.substring(lastIndex)
  const data = JSON.parse(chunk)
  // 更新最新一条 AI 消息
}
```

### 第二步：Backend 接收 HTTP 请求

`ai-chat-backend` 的 Gin 服务接收请求，经过中间件链处理：

1. **CORS 中间件**：设置跨域响应头
2. **鉴权中间件**（可选）：验证用户 Token
3. **限流中间件**：检查请求频率

然后路由到 `ChatService.ChatProcess` 控制器：

```go
// 控制器处理流程
func (s *ChatService) ChatProcess(ctx *gin.Context) {
    // 1. 解析请求
    var req ChatMessageRequest
    ctx.BindJSON(&req)
    
    // 2. 获取 gRPC 连接
    conn := grpcPool.Get()
    client := NewChatClient(conn)
    
    // 3. 构建请求参数
    chatReq := &ChatCompletionRequest{
        Message: req.Prompt,
        ChatParam: &config.Chat,
    }
    
    // 4. 发起 gRPC 流式调用
    stream, _ := client.ChatCompletionStream(ctx.Context, chatReq)
    
    // 5. 接收流式响应并转发给前端
    for {
        resp, err := stream.Recv()
        if err != nil {
            break
        }
        // 转换为 JSON 并通过 HTTP chunked 写出
        data, _ := json.Marshal(resp)
        ctx.Writer.Write([]byte(data + "\n"))
    }
}
```

### 第三步：Service 处理核心逻辑

`ai-chat-service` 接收 gRPC 请求后，执行以下处理：

#### 3.1 敏感词校验

```go
// 调用敏感词服务
ok, keyword := s.sensitive.Validate(userInput)
if !ok {
    // 命中敏感词，返回提示
    return "抱歉，我无法回答这个问题"
}
```

#### 3.2 关键词提取与向量检索

```go
// 提取关键词
keywords := s.keywords.FindAll(userInput)

// 在向量数据库中查找相似问题
similarQAs := s.vectorData.Search(keywords)

if len(similarQAs) > 0 {
    // 命中标准答案，直接返回
    return similarQAs[0].Answer
}
```

#### 3.3 获取对话上下文

```go
// 从 Redis 获取历史对话
contextMessages := s.chatContext.GetContext(userID, parentMessageID)
```

#### 3.4 调用大模型

```go
// 构建 OpenAI 请求
req := &openai.ChatCompletionRequest{
    Model:       "gpt-3.5-turbo",
    Messages:    contextMessages,
    MaxTokens:   2000,
    Temperature: 0.7,
    Stream:      true,
}

// 调用 OpenAI（通过 openai-api-proxy）
stream := client.ChatCompletionStream(req)
```

#### 3.5 保存聊天记录

```go
// 异步保存到 MySQL
go s.data.SaveChatRecord(userID, question, answer)

// 写入向量数据库供后续检索
go s.vectorData.SaveQuestion(question, keywords, answer)

// 更新 Redis 上下文
go s.chatContext.SaveContext(userID, question, answer)
```

### 第四步：流式返回响应

`ai-chat-service` 将大模型的流式响应转换为 gRPC 流消息：

```protobuf
message ChatCompletionStreamResponse {
  string id = 1;
  string object = 2;
  int64 created = 3;
  string model = 4;
  repeated ChatCompletionStreamChoice choices = 5;
}
```

`ai-chat-backend` 接收后转换为 HTTP chunked 响应，每条 JSON 以 `\n` 分隔：

```json
{"id":"...","choices":[{"delta":{"content":"你"}}]}
{"id":"...","choices":[{"delta":{"content":"好"}}]}
{"id":"...","choices":[{"delta":{"content":"！"},"finish_reason":"stop"}]}
```

### 第五步：前端展示

前端实时解析响应，将 `delta.content` 追加到当前 AI 消息中，实现边生成边显示的效果。用户可以看到类似打字机的输出体验。

## 五、技术选型说明

### 5.1 为什么采用微服务架构

本项目将系统拆分为多个独立服务，主要基于以下考虑：

- **职责分离**：各服务专注于单一职责，便于维护和迭代
- **独立部署**：每个服务可以独立部署和扩展，提高灵活性
- **技术多样**：不同服务可以使用最适合自己的技术栈
- **容错隔离**：单个服务故障不会影响整个系统

### 5.2 为什么采用 gRPC 作为内部通信协议

- **高性能**：gRPC 基于 HTTP/2，支持多路复用和流式传输
- **强类型**：通过 ProtoBuf 定义接口，自动生成类型安全代码
- **跨语言**：支持多种语言，便于各服务使用不同技术栈
- **流式支持**：原生支持服务器流式和双向流式

### 5.3 为什么采用 Redis 管理对话上下文

- **高性能**：内存数据库，读写速度快
- **TTL 支持**：天然支持设置过期时间
- **丰富数据结构**：支持多种数据结构，便于实现复杂逻辑

### 5.4 为什么需要向量数据库

- **语义检索**：支持相似问题匹配，不仅限于关键词匹配
- **知识库**：可以预先导入标准问答对，提供精准答案
- **扩展性**：可以不断扩充知识库，无需修改代码


## 八、常见问题与解决方案

### 8.1 前端无法连接后端

**检查项**：

1. 确认 `ai-chat-backend` 服务已启动
2. 检查 Nginx 代理配置是否正确
3. 确认前端 API 请求地址配置正确

### 8.2 聊天响应缓慢

**可能原因**：

1. OpenAI API 响应慢 → 可使用 openai-api-proxy 并检查上游速度
2. 网络延迟 → 检查服务间网络连接
3. 敏感词/关键词服务响应慢 → 检查服务负载和连接数

### 8.3 敏感词过滤不准确

**解决方案**：

1. 定期更新敏感词词库
2. 使用 `keywords-filter` 的格式化功能整理词库
3. 调整敏感词匹配算法参数

### 8.4 向量检索效果不佳

**优化方向**：

1. 扩充高质量问答知识库
2. 调整向量模型参数
3. 优化关键词提取逻辑

## 九、开发指南

### 9.1 本地开发启动顺序

```bash
# 1. 启动基础依赖
docker-compose up -d mysql redis vector-db

# 2. 启动敏感词/关键词服务
cd keywords-filter
go run ./filter-server/main.go --config=dev.config.yaml --dict=sensitive-dict.txt

# 3. 启动核心服务
cd ai-chat-service
go run ./chat-server/main.go --config=dev.config.yaml

# 4. 启动 HTTP 网关
cd ai-chat-backend
go run ./cmd/main.go --config=dev.config.yaml

# 5. 启动前端开发服务器
cd ai-chat-web
pnpm install
pnpm dev
```

### 9.2 添加新功能的典型流程

1. **在 proto 中定义接口** → 在 `ai-chat-service/proto` 中添加新的 gRPC 方法
2. **实现服务逻辑** → 在 `chat-server/server` 中实现具体业务
3. **添加 HTTP 接口** → 在 `ai-chat-backend/pkg/controllers` 中添加新的 Handler
4. **前端对接** → 在 `ai-chat-web/src/api` 中添加接口调用，在对应页面使用
