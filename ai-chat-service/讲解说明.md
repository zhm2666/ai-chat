## 模块概览

`ai-chat-service` 是整个 AI 聊天系统的**核心后端服务**，对外通过 gRPC 提供聊天能力，对内负责：

- **对接大模型（如 OpenAI）**，发起 ChatCompletion / 流式对话请求
- **管理对话上下文**，保证多轮对话能记住历史内容
- **敏感词过滤与关键词提取**，保障输出合规并支持知识检索
- **聊天记录与向量检索**，把问答落库并写入向量数据库，支持相似问答召回

上层的 `ai-chat-backend` 通过 gRPC 调用本模块，前端最终间接使用的也是这里提供的能力。

## 目录结构与职责说明

- **`chat-server/`**：gRPC 服务主入口与业务实现
  - `main.go`：服务启动入口
    - 加载配置（`dev.config.yaml`）
    - 初始化日志（`pkg/log`）
    - 初始化 MySQL / Redis / 向量数据库（`pkg/db/*`）
    - 创建 gRPC Server，注册 `Chat` 服务与健康检查
  - `server/server.go`：实现 `proto.ChatServer` 接口，是 AI 聊天的核心逻辑
    - `chatService.ChatCompletion`：一次性返回完整回答
    - `chatService.ChatCompletionStream`：流式返回回答（边生成边推送）
    - 内部会调用大模型、敏感词服务、关键词服务、向量库和数据库
  - `server/app.go`：封装一次聊天请求的上下文与工具
    - 负责把配置整理成 `openaiConf`
    - 创建 OpenAI 客户端（`getOpenaiClient`）
    - 构建大模型请求（`buildChatCompletionRequest`），控制 token、上下文长度等
    - 构建 proto 响应（`buildChatCompletionResponse` / `buildChatCompletionStreamResponse`）
    - 管理 Redis 上下文（`getContext` / `saveContext`）
    - 调用关键词 / 敏感词服务（`keywords` / `sensitive`）
  - `chat-context/`：基于 Redis 的对话上下文缓存
  - `data/`：聊天记录的数据库访问接口与实现（如持久化到 MySQL）
  - `vector-data/`：与向量数据库的交互（写入 / 查询向量化的聊天记录）

- **`services/`**：本服务依赖的外部服务客户端
  - `services.go`：公共辅助函数，例如 `AppendBearerTokenToContext` 为 gRPC 请求追加 Bearer Token
  - `grpc-client/` 与 `pkg/grpc_client_pool/`：gRPC 客户端连接池封装，复用到敏感词、关键词等依赖服务
  - `keywords-filter/`：
    - `proto/*.proto, *_grpc.pb.go`：敏感词 / 关键词服务的 gRPC 定义与生成代码
    - `keywords.go` / `sensitive.go`：对敏感词、关键词服务的业务封装
  - `tokenizer/`：计算消息在不同模型下的 token 数，用于控制上下文与响应长度

- **`pkg/`**：通用基础库
  - `config/`：配置加载
    - `Config` 结构体包含：
      - `Server`：服务监听地址、端口、访问 Token
      - `Mysql` / `Redis`：数据库和缓存配置
      - `Chat`：大模型调用相关配置（API Key、Base URL、模型、max_tokens、temperature 等）
      - `VectorDB`：向量数据库连接配置
      - `DependOn`：依赖的敏感词、关键词、分词服务地址与 Token
    - `InitConfig` / `GetConfig`：使用 Viper 加载 YAML 配置并提供全局访问
  - `db/`：数据库访问封装
    - `mysql`：初始化 MySQL，提供统一的 `GetDB`
    - `redis`：Redis 连接池与 key 前缀
    - `vector`：向量数据库的初始化与封装
  - `log/`：日志组件
    - 日志级别、输出位置（含日志滚动）、是否打印调用者
    - 为 `chatService`、`app` 等提供统一的 `ILogger` 接口
  - `constants/`：常量定义
  - `zerror/`：带错误码的错误封装
  - `utils/`：通用工具函数

- **`proto/`**：对外 gRPC 接口定义
  - `chat.proto`：定义了 `Chat` 服务及请求 / 响应消息结构
  - `chat.pb.go` / `chat_grpc.pb.go`：通过 `protoc` 生成的 Go 代码

- **根目录文件**
  - `dev.config.yaml`：开发环境配置
  - `README.md`：当前只包含生成 gRPC stub 和安装依赖的基础说明
  - `Dockerfile`：打包为容器镜像的构建脚本

## 请求处理流程（简化版）

从前端角度看，一次完整的聊天流程是：

1. 前端调用 `ai-chat-backend` 的 `/api/chat-process`
2. `ai-chat-backend` 通过 gRPC 调用本模块的 `ChatCompletionStream`
3. 本模块在 `chatService.ChatCompletionStream` 中处理请求：
   - 使用 `app.sensitive` 调用敏感词服务，判断是否允许回答
   - 使用 `app.keywords` 提取关键词，在向量库中查找相似问题并尝试命中标准答案
   - 如果没有命中，则调用 OpenAI（或兼容接口）进行补全
   - 同步或异步地：
     - 通过 `chat-context` 把问答对写入 Redis，用作后续上下文
     - 通过 `data` 把完整问答记录写入 MySQL
     - 通过 `vector-data` 把问答写入向量数据库，供之后相似问题召回
4. 把大模型的流式返回转换成 `ChatCompletionStreamResponse` 的多条消息，通过 gRPC 流推回 `ai-chat-backend`

一次性返回的 `ChatCompletion` 与上述流程类似，只是没有流式拆分。

## 使用到的主要设计模式

- **门面模式（Facade）**
  - `chatService` 对外提供统一的 gRPC 接口，内部封装了 DB、Redis、向量库、敏感词服务、大模型等复杂细节。
- **适配器模式（Adapter）**
  - `server/app.go` 中将内部的配置与 proto 请求，适配为 OpenAI SDK 所需的 `ChatCompletionRequest` 与消息结构。
- **仓储模式（Repository）+ 依赖注入**
  - `chat-server/data`、`chat-server/vector-data` 通过接口抽象数据访问，`NewChatServer` 通过构造函数注入具体实现，便于替换和测试。
- **对象池模式（Object Pool）**
  - `pkg/grpc_client_pool` 和 `services/grpc-client` 封装 gRPC 连接池，提高与外部服务通信的性能。
- **单例模式（Singleton）**
  - `pkg/config` 与部分 DB / 日志初始化通过包级变量提供全局唯一实例。

## 如何本地启动本模块（开发视角）

1. 在项目根目录进入本模块：

   ```bash
   cd ai-chat-service
   ```

2. 安装依赖（只需执行一次或偶尔更新时执行）：

   ```bash
   go mod tidy
   ```

3. 启动 gRPC 聊天服务（使用开发配置）：

   ```bash
   go run ./chat-server/main.go --config=dev.config.yaml
   ```

4. 确保相关依赖已准备好：
   - MySQL：用于存储聊天记录
   - Redis：用于存储对话上下文
   - 向量数据库：用于相似问答检索
   - 敏感词 / 关键词服务：由 `keywords-filter` 模块提供

本模块通常不直接对前端开放，而是通过 `ai-chat-backend` 的 HTTP 接口间接使用。

## 适合新手的阅读顺序建议

1. `chat-server/main.go`：了解服务是如何启动的
2. `chat-server/server/server.go`：跟着 `ChatCompletionStream` 看一次完整的聊天处理流程
3. `chat-server/server/app.go`：看如何构造 OpenAI 请求、管理上下文与敏感词 / 关键词
4. `pkg/config/config.go` 与 `dev.config.yaml`：了解配置项的含义，方便本地调试
5. `services/keywords-filter` 与 `chat-server/vector-data`：作为进阶，理解敏感词与向量检索是如何接入的

掌握以上内容后，你就基本理解了 `ai-chat-service` 如何作为“AI 大脑”为整个系统提供聊天能力。
