## 模块概览

`keywords-filter` 是整个系统中的 **关键词 / 敏感词过滤服务** 模块，使用 Go + gRPC 实现，主要职责是：

- 提供统一的 gRPC 接口，对外暴露：
  - 文本敏感词校验（是否命中敏感词、命中的词是什么）。
  - 文本关键词提取（返回命中的所有关键词列表）。
- 基于本地词库文件（如 `sensitive-dict.txt`、`keyword-dict.txt` 等）构建高性能过滤器。
- 为 `ai-chat-service` 等上游服务提供合规检查与关键词抽取能力。

## 目录结构与职责说明

- **根目录**
  - `README.md`：
    - 说明如何通过 `protoc` 生成 gRPC stub。
    - 说明如何执行基准测试与性能分析。
    - 给出启动脱敏（敏感词过滤）服务的示例命令：
      ```bash
      go run .\filter-server\main.go --config=dev.config.yaml --dict sensitive-dict.txt
      ```
  - `go.mod` / `go.sum`：Go 依赖管理文件。
  - `dev.config.yaml`：开发环境配置（服务监听地址、端口、访问 Token、日志配置等）。
  - `dict.txt` / `sensitive-dict.txt` / `keyword-dict.txt`：
    - 存放敏感词或关键词的本地字典文件。
    - 在启动时通过 `--dict` 参数指定加载的词库文件。
  - `Dockerfile`：将本服务打包为容器镜像的构建脚本。

- **`proto/`**
  - `filter.proto`：定义过滤服务的 gRPC 接口与消息结构，主要包含：
    - `FilterReq`：请求消息，包含待检测的文本 `text`。
    - `ValidateRes`：校验结果（`ok` 标识是否通过，`keyword` 为命中的敏感词）。
    - `FindAllRes`：包含所有命中关键词的数组。
    - `service Filter`：定义 `Validate` 和 `FindAll` 两个 RPC 方法。
  - `filter.pb.go` / `filter_grpc.pb.go`：由 `protoc` 生成的 Go 代码。

- **`filter-server/`**
  - `main.go`：gRPC 服务入口。
    - 解析命令行参数：
      - `--config`：指定配置文件（默认 `dev.config.yaml`）。
      - `--dict`：指定词库文件路径（默认 `dict.txt`）。
      - `--format`：是否对词库文件进行格式化（去重 / 规范化）。
    - 加载配置（`pkg/config`），初始化日志（`pkg/log`）。
    - 调用 `filter.InitFilter(dictFilePath)` 加载词库，构建过滤器实例。
    - 启动 gRPC Server：
      - 监听在 `Server.IP:Server.Port`。
      - 注册拦截器（`filter-server/interceptor`）用于鉴权等。
      - 注册 `Filter` 服务实现与健康检查服务。
  - `server/server.go`：gRPC 服务实现。
    - `filterService` 结构，实现 `proto.FilterServer` 接口。
    - `NewFilterService(filter.IFilter)`：通过依赖注入获取底层过滤器。
    - `Validate`：
      - 调用底层 `filter.Validate` 对文本进行敏感词检测。
      - 返回 `ValidateRes{ Ok: ok, Keyword: word }`。
    - `FindAll`：
      - 调用底层 `filter.FindAll` 获取全部命中关键词。
      - 返回 `FindAllRes{ Keywords: words }`。
  - `interceptor/auth.go`：
    - gRPC 拦截器，通常用于校验 Metadata 中的 Token 等，确保只有授权的上游服务能调用本服务。

- **`pkg/filter/`**
  - `filter.go`：核心过滤器封装。
    - 使用第三方库 `github.com/importcjj/sensitive` 作为底层敏感词引擎。
    - 定义接口：
      ```go
      type IFilter interface {
        Validate(text string) (bool, string)
        FindAll(text string) []string
      }
      ```
    - 实现结构体 `filter`：
      - 内部持有 `*sensitive.Filter`。
      - `Validate`：
        - 对输入文本进行首尾空格处理并在两端补空格，避免分词边界问题。
        - 调用 `sensitive.Filter.Validate` 返回是否命中与命中的词。
        - 对命中的词进行 `strings.Trim` 去除多余空格。
      - `FindAll`：
        - 同样先标准化文本。
        - 调用 `sensitive.Filter.FindAll` 返回所有命中词，并统一去除首尾空格。
    - 全局管理：
      - `_filter`：包级变量，存放当前全局 `filter` 实例。
      - `GetFilter()`：返回全局 `IFilter` 实例。
      - `InitFilter(dictFilePath string)`：
        - 校验词库文件路径是否存在。
        - 调用 `sensitive.New()` 创建引擎实例。
        - 设置 `UpdateNoisePattern("")`（噪声模式为空）以避免误过滤。
        - 加载词库文件 `LoadWordDict(dictFilePath)`。
  - 字典处理：
    - `OverwriteDict(dictFilePath string)`：
      - 用于重新格式化词库文件，主要包括：
        - 遍历原始文件，对每行词条去重。
        - 使用正则 `\p{Han}+` 判断是否包含中文。
        - 对纯非中文词条在前后加空格，便于过滤时减少误伤。
        - 将整理后的内容写回同一个文件。

- **`pkg/config/`**
  - `config.go`：
    - 使用 Viper 加载配置文件，映射到：
      - `Server`：`IP`、`Port`、`AccessToken`。
      - `Log`：日志级别、日志路径。
    - 提供 `InitConfig` 与 `GetConfig` 作为全局访问入口。

- **`pkg/log/`**
  - 封装日志组件：
    - 支持日志级别配置与日志文件滚动。
    - 提供 `NewLogger`、`SetLevel`、`SetOutput`、`SetPrintCaller` 等函数。

- **`filter_test/`**
  - `filter_test.go`：
    - 定义了对 `pkg/filter` 的基准测试（`go test --bench`）。
    - 用于评估在不同 CPU 配置下的性能表现。

- **`client/`**
  - `main.go`：
    - 简单的 gRPC 客户端示例，用来测试连接过滤服务并发送请求。

## 对外提供的功能（gRPC 接口）

通过 `proto.Filter` 服务，本模块对外提供两个主要功能：

1. **敏感词校验（Validate）**
   - 请求：`FilterReq{ text: "要检测的文本" }`
   - 响应：`ValidateRes{ ok: bool, keyword: string }`
   - 行为：
     - 如果 `ok == true`：表示文本通过校验，没有命中敏感词。
     - 如果 `ok == false`：表示文本不通过，`keyword` 是被命中的敏感词。

2. **关键词提取（FindAll）**
   - 请求：`FilterReq{ text: "要检测的文本" }`
   - 响应：`FindAllRes{ keywords: []string }`
   - 行为：
     - 返回所有在词库中出现的词，适用于：
       - 关键词检索（如在向量库中查相似问答）。
       - 文本打标签等场景。

在整个系统中，`ai-chat-service` 会通过 gRPC 调用这两个接口：

- 在用户输入时调用 `Validate` 检测是否触发敏感词，不通过则直接给出「知识盲区」提示，而不调用大模型。
- 在用户输入时调用 `FindAll` 提取关键词，用于向量检索命中已有 “标准答案”。

## 使用到的主要设计模式

- **门面模式（Facade）**
  - `filterService` 对外只暴露简单的 `Validate` 和 `FindAll`，内部隐藏了词库加载、正则处理、底层引擎等复杂细节。

- **适配器模式（Adapter）**
  - `pkg/filter` 封装第三方库 `github.com/importcjj/sensitive`，对外通过自定义的 `IFilter` 接口暴露统一能力，适配不同调用方。

- **单例模式（Singleton）**
  - `pkg/config` 和 `pkg/filter` 都通过包级变量维护单例实例（配置与过滤器），在整个服务生命周期内复用。

- **依赖注入（Constructor Injection）**
  - `server.NewFilterService(filter.IFilter)` 使用构造函数注入底层过滤器实现，便于测试与替换。

## 本地启动与调试步骤

1. 进入模块目录：

   ```bash
   cd keywords-filter
   ```

2. 安装依赖：

   ```bash
   go mod tidy
   ```

3. 启动过滤服务（使用开发配置与示例词库）：

   ```bash
   go run ./filter-server/main.go --config=dev.config.yaml --dict=sensitive-dict.txt
   ```

4. 使用 gRPC 客户端或项目中的 `client/main.go` 进行简单测试：

   ```bash
   go run ./client/main.go
   ```

5. 需要格式化词库（去重、规范化）时，可以使用：

   ```bash
   go run ./filter-server/main.go --dict=sensitive-dict.txt --format=true
   ```

6. 性能测试与分析（可选）：

   ```bash
   go test --bench . ./filter_test
   go test --bench FindAll --run ^$ -cpu 1 -outputdir ./ -cpuprofile cpu.out ./filter_test
   go tool pprof -web cpu.out
   ```

1. `README.md`：先了解如何生成 gRPC stub、如何启动服务与做基准测试。

掌握这些内容后，你就能比较自信地修改词库、调整过滤逻辑，或者在其他服务中正确地调用 `keywords-filter` 模块提供的 gRPC 接口。
