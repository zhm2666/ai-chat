## 模块概览

`ai-chat-backend` 是整个 AI 聊天系统的 **HTTP 网关 / BFF（Backend For Frontend）** 模块，主要职责是：

- 对前端（`ai-chat-web`）暴露统一的 HTTP 接口，例如 `/api/chat-process`。
- 把前端的 HTTP+JSON 请求转换为 gRPC 请求，转发到 `ai-chat-service`。
- 接收 `ai-chat-service` 返回的流式 gRPC 响应，再转换为前端可消费的 HTTP 流。
- 提供基础中间件能力（CORS、鉴权、限流），以及静态页面托管能力。

前端不会直接访问 gRPC 服务，而是全部经过本模块，从而在这一层集中做协议适配、安全控制和网关逻辑。

## 目录结构与职责说明

- **`cmd/`**
  - `main.go`：服务启动入口。
    - 解析命令行参数（如 `--config=dev.config.yaml`）。
    - 通过 `pkg/config` 加载配置（HTTP 监听地址、依赖服务地址、模型参数等）。
    - 初始化日志（`pkg/log`）。
    - 构造并运行 `ChatGPTWebServer`，启动 Gin HTTP 服务。

- **`pkg/`**
  - `config/`
    - `config.go`：使用 Viper 从配置文件（如 `dev.config.yaml`）加载并反序列化到 `Config` 结构体。
      - `Http`：HTTP 服务的 IP 和端口。
      - `Log`：日志级别与日志路径。
      - `DependOn`：
        - `AiChatService`：gRPC AI 服务地址和访问 Token。
        - `User`：用户中心服务地址（用于鉴权）。
      - `Chat`：与模型相关的默认参数（模型名、max_tokens、temperature、上下文长度等），会透传给 `ai-chat-service`。
    - 提供 `InitConfig` / `GetConfig` 两个函数，采用全局单例方式供其他包使用。
  - `controllers/`
    - `chat.go`：核心控制器，负责处理前端的 `/api/chat-process` 请求。
      - 定义请求/响应结构：
        - `ChatMessageRequest`：包含 `prompt` 和 `options`（如 `parentMessageId`）。
        - `ChatMessage`：返回给前端的消息结构，包含 `id`、`text`、`delta`、`detail` 等。
      - `NewChatService`：构造 `ChatService`，注入配置与日志。
      - `ChatService.ChatProcess`：
        1. 解析前端传来的 JSON 请求。
        2. 从 gRPC 客户端池中获取与 `ai-chat-service` 的连接。
        3. 组装 `ChatCompletionRequest`，填入模型配置、上下文参数等。
        4. 发起 gRPC 流式调用 `ChatCompletionStream`。
        5. 循环接收 gRPC 流消息，将其转换为 `ChatMessage`，序列化为 JSON。
        6. 通过 HTTP chunked 方式一条一条写回给前端（每条为一行 JSON）。
  - `middlewares/`
    - `cors.go`：跨域中间件。
      - 设置 `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods` 等头，处理 OPTIONS 预检请求。
    - `auth.go`：鉴权中间件（默认在路由里是注释掉的，需要时可以启用）。
      - 从 `Authorization` Header 中解析 Bearer Token。
      - 调用用户中心服务（`DependOn.User.Address`）校验 Token。
      - 在 Gin 上下文中设置用户信息（`User.ID`、`User.Name` 等），用于后续业务。
    - `rate_limit.go`：简单的单节点限流中间件。
      - 基于 `golang.org/x/time/rate` 实现固定速率限流。
      - 超出速率时返回 429 状态码和统一 JSON 结构。
  - `log/`
    - 日志封装，包括日志级别、输出（含滚动）、是否打印调用者信息等。
    - 提供 `ILogger` 接口，供控制器和入口代码统一使用。
  - `tokenizer/`、`utils/`、`zerror/` 等：
    - 提供一些工具函数、错误封装和可能的 token 相关辅助（具体可按需阅读）。

- **`services/`**
  - `ai-chat-service/`
    - `ai_chat_service.go`：封装调用下游 `ai-chat-service` 的逻辑，通常包含：
      - gRPC 连接池的获取与管理。
      - 生成 `ChatClient`，并在其中复用连接。
    - `proto/`：`ai-chat-service` 模块的 gRPC proto 与生成代码，用于构建 gRPC 客户端。
  - `grpc-client/`
    - `grpc_client_pool.go` 等：封装通用的 gRPC 客户端连接池逻辑，提升连接复用和性能。
  - `services.go`：
    - 提供辅助函数，如 `AppendBearerTokenToContext`：在 gRPC metadata 中追加 `Authorization: Bearer <token>`。

- **`www/`**
  - 编译后的前端静态资源（`index.html` + JS/CSS）。
  - 在 `cmd/main.go` 中通过 `http.FileServer` 托管：
    - `/`：返回 `www/index.html`。
    - 未匹配的路径走 `NoRoute`，统一由前端路由接管。

- **根目录其他文件**
  - `dev.config.yaml`：本模块的开发环境配置文件。
  - `Dockerfile`：打包为容器镜像的构建脚本。
  - `go.mod` / `go.sum`：Go 依赖管理。

## 请求处理流程（从前端到 AI 服务）

一次完整的聊天请求大致流程如下：

1. 前端向本模块发起 HTTP 请求：
   - `POST /api/chat-process`
   - 请求体 JSON 示例：
     ```json
     {
       "prompt": "你好",
       "options": {
         "name": "default",
         "parentMessageId": ""
       }
     }
     ```
2. Gin 将请求路由到 `ChatService.ChatProcess`。
3. `ChatProcess`：
   - 使用 `ctx.BindJSON` 解析请求到 `ChatMessageRequest`。
   - 生成一个新的 `messageID`（`uuid.New().String()`）。
   - 从 gRPC 客户端池中 `Get()` 一个连接，构建 `ChatClient`。
   - 读取本地配置里的模型参数（`config.Chat`），填入 `ChatParam`。
   - 通过 `services.AppendBearerTokenToContext` 在 Context 中加入访问 Token。
   - 调用下游 gRPC 方法 `ChatCompletionStream` 获取流式响应。
4. 在收到 gRPC 流中的每一条消息时：
   - 更新 `ChatMessage` 结构体（包括 `ID`、`Delta`、`Text` 累积内容、`Detail` 等）。
   - 序列化为 JSON 字节。
   - 通过 HTTP 响应流写出到前端：
     - Content-Type 为 `application/octet-stream`。
     - 多条 JSON 使用换行符分隔（每行一个 JSON），前端按行解析。
5. 前端收到每一行 JSON 后，实时将 `delta` 或累积的 `text` 展示在聊天窗口，实现「边生成边显示」的体验。

## 使用到的主要设计模式

- **门面模式（Facade）**
  - 整个模块对前端提供统一的 HTTP 接口（`/api/chat-process`），对下游隐藏了 gRPC、Token、模型参数、连接池等细节。

- **适配器模式（Adapter）**
  - 在 `ChatService.ChatProcess` 中，把前端的 HTTP+JSON 请求映射为 gRPC `ChatCompletionRequest`。
  - 同时把 gRPC 的流式响应适配成前端定义的 `ChatMessage` JSON 结构。

- **中间件 / 装饰器模式（Decorator）+ 责任链模式（Chain of Responsibility）**
  - Gin 的中间件（`Cors`、`Auth`、`RateLimitMiddleware` 等）围绕核心 handler 执行前后逻辑：
    - CORS 设置头。
    - 鉴权检查用户信息。
    - 限流控制请求频率。
  - 多个中间件串联形成一条责任链，按顺序处理请求。

- **对象池模式（Object Pool）**
  - 在 `services/grpc-client` 与 `services/ai-chat-service` 中使用 gRPC 客户端连接池，复用 `*grpc.ClientConn`，减少频繁建连带来的性能损耗。

- **单例模式（Singleton）**
  - `pkg/config`、`pkg/log` 使用包级变量维护全局唯一实例，便于在各处直接获取配置与日志实例。

## 如何本地启动本模块（开发视角）

1. 进入模块目录：

   ```bash
   cd ai-chat-backend
   ```

2. 安装依赖：

   ```bash
   go mod tidy
   ```

3. 使用开发配置启动 HTTP 服务：

   ```bash
   go run ./cmd/main.go --config=dev.config.yaml
   ```

4. 确保依赖服务可用：
   - `ai-chat-service`：gRPC AI 服务，地址与 Token 在 `dev.config.yaml` 的 `dependOn.ai-chat-service` 中配置。
   - 用户服务（可选）：若要启用 `Auth` 中间件，需要配置并启动用户中心服务。

5. 启动成功后，你可以：
   - 直接访问健康检查接口：`GET /api/health`。
   - 配合前端 `ai-chat-web`，通过 `/api/chat-process` 完成一次完整的 AI 聊天请求。

## 适合新手的阅读顺序建议

1. `cmd/main.go`：了解服务是如何启动、如何注册路由和中间件的。
2. `pkg/config/config.go` + `dev.config.yaml`：搞清楚 HTTP 端口、依赖服务地址、模型参数从哪里来。
3. `pkg/controllers/chat.go`：详细阅读 `ChatProcess`，理解“HTTP 请求 → gRPC 调用 → HTTP 流响应”的完整链路。
4. `pkg/middlewares/{cors.go, auth.go, rate_limit.go}`：了解跨域、鉴权、限流是如何插入到请求处理链路中的。
5. `services/ai-chat-service` 与 `services/grpc-client`：作为进阶，理解 gRPC 连接池与下游服务调用的细节。

掌握这些内容之后，你就可以比较自信地在 `ai-chat-backend` 中新增接口、调整中间件，或修改与 `ai-chat-service` 交互的逻辑了。
